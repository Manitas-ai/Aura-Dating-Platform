Executive Summary

Aura is a proof-of-concept dating platform built to demonstrate the feasibility of a premium, feature-complete matchmaking product using modern web technologies and a serverless backend. The platform operates two independent applications — a consumer-facing experience and a management console — both connected to a shared Supabase (Postgres) backend with no custom server infrastructure required.

The PoC supports the full core loop: member discovery, mutual matching via automated database triggers, and in-app messaging. A lightweight username/password authentication mechanism enables multi-user simulation without the overhead of a production identity system. Ten pre-seeded member profiles, five established matches, and seeded conversation threads are available for immediate demonstration.

01 Applications

Two dedicated interfaces are served from a single GitHub repository, each deployed as an independent Netlify site.

Aura — User Application /user-app

The consumer-facing experience. Members log in, browse the discovery queue, express interest, and communicate with mutual connections — all within a premium dark-mode interface optimised for mobile use.

  • Username and password login with persistent session via localStorage
  • Discover page: single-card profile stack with Like / Pass interactions
  • Profile detail modal: full biography, interest tags, photo
  • Matches page: grid of mutual connections with unread indicators
  • In-app messaging: per-match conversation threads with sent/received layout
  • My Profile: view and edit bio, occupation, location, interest tags
Aura — Admin Console /admin-app  🔒

The operator management interface. Access is restricted by a single admin password. Provides complete visibility and control over platform activity — members, matches, and message moderation.

  • Password-gated entry — no shared credentials with user application
  • Dashboard: KPI cards (members, new today, matches, messages)
  • Members table: search, filter by status, edit profile fields, suspend/reactivate, delete
  • Matches overview: all mutual connections with member photos and dates
  • Messages moderation: expandable thread view, per-message deletion capability
02 Authentication Model

A deliberately simplified auth mechanism enables multi-persona simulation without a production identity system. Not intended for production use.

Each member profile stores a username (unique text) and a password (plaintext, max 5 characters) directly in the profiles table. On login, the application performs a single SELECT query against these fields. On success, the matching profile.id and profile data are persisted to localStorage under the key aura_profile_id. Logout clears this entry. This allows the operator to switch between any of the ten pre-built personas and experience the platform from each member's perspective — including their specific likes, matches, and message history.

Selected for developer velocity, zero server overhead, and straightforward path to production scaling.

Auto-match trigger: A PL/pgSQL function (create_match_if_mutual()) fires AFTER INSERT on likes. It checks for a reciprocal like and, if found, inserts a row into matches using LEAST()/GREATEST() to canonicalise the pair order and prevent duplicates via a UNIQUE constraint.
