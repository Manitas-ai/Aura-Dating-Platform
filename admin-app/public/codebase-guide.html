🗺️ The Big Picture — What Is This Project?

Aura is a dating app. Like Tinder, but more thoughtful — it is built on deep personality questionnaires, not just photos. The codebase is split into two separate apps that share one database.

👤

User App

What real users see. Browse profiles, send flirts, chat, manage your own profile. Lives in the user-app/ folder.

🛠️

Admin App

What the owner/moderator sees. Dashboard stats, manage members, read messages, view feedback. Lives in the admin-app/ folder.

🗄️

Supabase (Database)

The shared brain in the cloud. Stores all users, messages, photos, flirts. Both apps read and write to it.

📄

setup.sql

A script that creates the database structure and fills it with 10 test users to demo the app.

🏠 REAL WORLD ANALOGY

Think of a restaurant. The user-app is the dining room where customers eat. The admin-app is the manager's office. Supabase is the kitchen where all the food (data) is stored and prepared. Both rooms connect to the same kitchen.

🧰 The Technologies — What Are They?

⚛️

React

A JavaScript library for building user interfaces. Instead of writing one giant HTML page, you write small reusable components (like Lego bricks) that React assembles into a full page. Files end in .tsx.

🔷

TypeScript

JavaScript with "labels" on everything. Instead of a variable that could be anything, you say "this must be a string" or "this must be a number". It catches mistakes before they become bugs. Files end in .ts or .tsx.

🎨

Tailwind CSS

A way to style your app by adding small class names directly in your HTML, like bg-blue-500 for a blue background or text-lg for large text. No separate CSS files needed for most styling.

🗄️

Supabase

A cloud database service. It gives you a PostgreSQL database, file storage (for photos), and an API — all in one. You connect to it from your React code using a tiny library.

⚡

Vite

The build tool. It takes all your TypeScript/React files and bundles them into plain HTML + JavaScript that a browser can understand. Also runs a fast local development server.

🔁

React Router

Makes the app feel like a multi-page website even though it's really one HTML file. When you click "Discover", it swaps the content without reloading the whole page. This is called a Single Page App (SPA).

📚 HOW THEY WORK TOGETHER

TypeScript is your language. React is your UI framework. Tailwind makes it look good. Vite compiles it all. Supabase stores the data. Think of TypeScript as the pen, React as the paper, Tailwind as the paintbrush, Vite as the printer, and Supabase as the filing cabinet.

💡 WHAT IS A .tsx FILE?

It stands for TypeScript + JSX. JSX lets you write HTML-like code inside JavaScript. So a .tsx file is TypeScript code that contains your component logic AND its HTML structure in the same file.

🖥️ Frontend vs Backend — What's the Difference?

🖥️

Frontend FE

Everything the user sees and clicks. Runs in the browser on the user's device. All the React, TypeScript, and Tailwind files are frontend. They create the buttons, forms, colours, layouts.

🖧

Backend BE

Everything that stores and processes data. Runs on a server, not in the browser. In this project, the backend is entirely Supabase — the database tables, the photo storage, and the SQL queries.

🍽️ REAL WORLD ANALOGY

The frontend is the menu and the waiter. It's what you see and interact with. The backend is the kitchen — you never see it, but it's where the food (data) actually gets made and stored. When you click "Send Message", the frontend (waiter) takes your order to the backend (kitchen) which saves it to the database, then brings back a confirmation.

💡 THIS PROJECT HAS NO CUSTOM BACKEND SERVER

There's no Node.js or Python server written here. Supabase provides the backend as a service. The React app talks directly to Supabase using its JavaScript library. This is common for small apps and prototypes.

👤 User App — File by File

These are the key files in user-app/src/. Hover over any card to see more detail.

main.tsx
React TypeScript FRONTEND
The very first file that runs. It finds the <div id="root"> in index.html and injects the whole React app into it. Also wraps the app in AuthProvider (so all pages know who is logged in) and BrowserRouter (so navigation works).
Think of it as the "light switch" — flipping it on starts everything. You almost never need to edit this file.
App.tsx
React FRONTEND
The router. It maps URL paths to page components. /discover → DiscoverPage, /flirts → FlirtsPage, etc. Also handles redirects: if you are not logged in and try to visit /discover, it sends you to /login.
Uses React Router v6. The <Route> tags are like a table of contents for the app.
index.css
Tailwind CSS FRONTEND
Global stylesheet. The top 3 lines (@tailwind base/components/utilities) activate Tailwind. Below that are custom CSS classes like .btn-gold and .card that are reused throughout the app. Also imports Google Fonts.
Tailwind generates hundreds of utility classes (bg-*, text-*, p-*, etc.) from these 3 lines. The custom classes at the bottom are shortcuts for commonly repeated groups of Tailwind classes.
context/AuthContext.tsx
React Context FRONTEND
The login memory of the app. After you log in, your profile data is saved here (and in localStorage). Any component in the app can then call useAuth() to know who is logged in — without having to pass it as props through every component manually.
React Context is like a global variable that all components can read. Without it, you'd have to pass "who is logged in" as a prop from App → Layout → Page → Component → SubComponent, which gets messy fast.
lib/supabase.ts
Supabase BRIDGE
The database connection. These ~5 lines of code create a connection to the Supabase cloud database using a URL and an API key. The result is exported as db, which every page imports to query the database.
Example usage in a page: const { data } = await db.from('profiles').select('*') — this reads all rows from the profiles table. The Supabase library handles all the HTTP requests behind the scenes.
lib/constants.ts
TypeScript FRONTEND
A large list of all the dropdown options used in questionnaires and filters — 32 DACH regions, age groups, lifestyle choices, diet options, and more. Centralised here so every page uses the same values.
Instead of typing "Bavaria" in 5 different places (and possibly making a typo), you define it once here and import it everywhere.
types/index.ts
TypeScript FRONTEND
The blueprints. Defines what shape the data must have. A Profile must have a username (string), an age_group (string), etc. TypeScript uses these to warn you if you try to use a field that doesn't exist.
Analogy: a blueprint for a house. You define "a house has a door, 2 windows, and a roof". If you try to build a house with 3 doors but no roof, TypeScript immediately flags the error before you even run the code.
components/Layout.tsx
React Tailwind FRONTEND
The navigation shell. The top bar on desktop (with logo and links) and the bottom tab bar on mobile. It wraps all the authenticated pages. Uses React Router's <Outlet /> — a placeholder that renders whichever page is currently active.
This is how you avoid repeating the navigation code in every single page. One Layout wraps them all.
pages/LoginPage.tsx
React Supabase FRONTEND
The login form. When you submit, it queries the profiles table in Supabase: "does a user exist with this username and password?". If yes — stores the profile in AuthContext and redirects to /discover. Also logs the login to the user_logins table.
Note: passwords are stored as plain text (max 5 chars). This is fine for a prototype/demo but would need encryption (bcrypt/hashing) in a real production app.
pages/RegisterPage.tsx
React Supabase FRONTEND
A multi-step signup wizard with 10 steps. Step 1 collects basic info. Steps 2–10 walk through the personality questionnaire. At the end, it inserts a new row into profiles AND into questionnaires in Supabase.
The progress bar is calculated as (currentStep / totalSteps) * 100. Each step uses React state to save what you typed, building up an object that gets submitted all at once at the final step.
pages/DiscoverPage.tsx
React Supabase FRONTEND
The browse screen. Fetches all active profiles from Supabase, then uses JavaScript's useMemo to filter them by gender, age, region, etc. without re-fetching. Displays one profile at a time. Clicking "Observe" inserts a row into the observations table.
useMemo means "compute this filtered list, but only when the filters change — not every time the page re-renders". This is a performance optimisation.
pages/FlirtsPage.tsx
React Supabase FRONTEND
The messaging screen. Two tabs: active chats (accepted flirts) and pending proposals. Selecting a conversation loads messages from the messages table. Typing and sending inserts a new message. Accepting a proposal inserts a row into the flirts table.
On mobile it shows either the list OR the chat (not both). On desktop it shows both side by side. This is achieved with Tailwind's responsive classes like hidden md:block.
pages/MyProfilePage.tsx
React Supabase FRONTEND
Edit your profile and manage photos. Uploading a photo sends it to Supabase Storage (a file hosting service). The returned URL is saved in the profile_photos table. You can upload up to 5 photos, set one as primary, and delete others.
Supabase Storage works like a mini Google Drive in the cloud. You upload a file, get back a public URL, then store that URL in the database.

🛠️ Admin App — File by File

Files in admin-app/src/. Same structure as user-app, different purpose.

context/AuthContext.tsx (admin)
React Context FRONTEND
Admin-only password gate. The password is hardcoded as aura2024 (no database). When entered correctly, it stores true in localStorage. All admin pages check this — if not logged in, redirect to the admin login page.
components/AdminLayout.tsx
React Tailwind FRONTEND
The dark sidebar with navigation links to all admin sections. Uses React Router's <NavLink> which automatically adds an "active" style to the currently selected menu item.
pages/DashboardPage.tsx
React Supabase FRONTEND
The home screen for the admin. Shows 4 KPI cards: total members, joined today, active flirts, messages sent. Each number is fetched from Supabase. Also shows a "recent members" list.
pages/MembersPage.tsx
React Supabase FRONTEND
A full table of all users with search and filter. The admin can edit user details (UPDATE in Supabase), change their status to suspended/active, or delete them (DELETE from Supabase). This is the main moderation tool.
pages/MessagesPage.tsx
React Supabase FRONTEND
Lists all message threads from all users. The admin can expand any thread to read the messages and delete individual messages if needed. This is the content moderation page.
pages/UserLoginsPage.tsx
React Supabase FRONTEND
Shows the last 500 login events from the user_logins table — who logged in and when. Every time a user successfully logs in via the user-app, a row is inserted there automatically.

🗄️ Supabase — The Database Explained

Supabase is the only backend in this project. It runs in the cloud. You manage it via a browser dashboard at supabase.com. The setup.sql file creates all these tables when pasted into the Supabase SQL editor.

💡 WHERE IS SUPABASE CONNECTED IN CODE?

In both user-app/src/lib/supabase.ts and admin-app/src/lib/supabase.ts. These files contain the Supabase project URL and public API key. Every page that needs data imports db from this file.

The 9 Database Tables

profiles
The main user table. Stores username, password, age group, gender, region, about me text, interests list, and photo URL. Also has a status field (active/suspended) that admins can change.
Used by: LoginPage, RegisterPage, DiscoverPage, MembersPage (admin)
questionnaires
Deep personality data for each user. 28 fields covering life situation, physical appearance, lifestyle, values, dealbreakers, and intellectual interests. Also has a q_visibility JSON field — users can set each chapter to "public" or "flirts only".
Used by: RegisterPage, QuestionnaireEditPage, ProfileDetailModal
profile_photos
Multiple photos per user. Stores the photo URL (from Supabase Storage), whether it's the primary photo, and the display order. Each user can have up to 5 photos.
Used by: MyProfilePage
observations
Your private watchlist. When you click "Observe" on a profile, a row is added here. The observed person never knows. You can send a flirt proposal to anyone in your observations list.
Used by: DiscoverPage (add), ObservePage (view/remove)
flirt_proposals
The first step of connecting. When you like someone, you send a proposal with an optional personal message. They can accept or decline. You get max 2 attempts per person. Status: pending / accepted / declined.
Used by: ObservePage (send), FlirtsPage (accept/decline)
flirts
A mutual connection — created only when someone accepts a proposal. This is the "match" equivalent. Once a flirt exists, both users can message each other.
Used by: FlirtsPage (view/create), MatchesPage (admin view)
messages
Chat messages between two connected users. Each message belongs to a flirt (connection). Stores sender ID, content, and timestamp.
Used by: FlirtsPage (send/receive), MessagesPage (admin moderate)
user_logins
A login history log. Every time a user logs in successfully, a new row is inserted with their username and a timestamp. Used by admins to track activity.
Used by: LoginPage (insert), UserLoginsPage admin (view)
feedback
User feedback submissions — category (bug, idea, etc.), a 1–5 star rating, and free text. The admin can filter and read these in the FeedbackPage admin.
Used by: FeedbackPage user (submit), FeedbackPage admin (read)

Supabase Storage (for Photos)

profile-photos (Storage Bucket)
Not a database table — this is a file storage bucket, like Dropbox. When a user uploads a profile photo, the image file goes here. Supabase returns a public URL. That URL is then saved in the profile_photos table.
Used by: MyProfilePage (upload/delete)

📦 THE GENERAL DATA FLOW PATTERN

Browser (React renders UI) ↔ Supabase JS library (translates your code into HTTP requests) ↔ Supabase cloud (runs your query) ↔ PostgreSQL (stores the data). The React app is the client — it always asks Supabase. Supabase is the server — it always has the data.

📐 How a React Component Works (Simple Example)

📖 READ THIS IF YOU'RE NEW TO REACT

A React component is just a JavaScript function that returns HTML (called JSX). It can have state (data that changes) and effects (code that runs when the page loads or when something changes).

// Simplified version of what a page looks like
function DiscoverPage() {
// STATE: data that can change and triggers a re-render
const [profiles, setProfiles] = useState([]) ← list of users
// EFFECT: runs once when the page loads
useEffect(() => {
db.from('profiles').select('*') ← fetch from Supabase
.then(({ data }) => setProfiles(data)) ← save to state
}, [])
// RENDER: the HTML this component shows
return (
<div className="grid gap-4"> ← Tailwind class
{profiles.map(p => <ProfileCard key={p.id} profile={p} />)} ← React component
</div>
)
}

🧩 Key Concepts Summary

State (useState)

Data stored inside a component that, when changed, causes React to re-render the screen. Example: the list of profiles, the current filter value, the text you're typing.

Effect (useEffect)

Code that runs "as a side effect" — usually to fetch data from Supabase when a page first loads, or when a dependency (like a filter) changes.

Context (useContext)

Global state shared across all components. Used here for "who is logged in". Instead of passing it as props everywhere, any component can just call useAuth().

Props

Data passed from a parent component to a child component. Like function arguments. Example: <ProfileCard profile={p} /> passes the profile object as a prop.

Component

A reusable UI building block. A Button, a Card, a Modal — each is a component. Compose them like Lego to build pages. Defined once, used many times.

Route

A mapping from a URL path to a component. /discover renders DiscoverPage, /profile renders MyProfilePage. React Router handles this without page reloads.
